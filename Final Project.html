Final Project

實驗動機

  • 回顧 HW 5, 我們已經了解 unperfect transmission 發生的原因以及其對於傳輸結果的影響.

  • 由於如Video Streaming等應用對 delay 和 loss 有較高的要求 , 但現實生活中的網路傳輸卻無法避免delay和loss的發生。因此，如何於存在delay和loss的網路中藉由傳輸機制來提升網路傳輸的效果，是這個Final Project要探討的議題。

  • Final Project:

    • 目標: Server可以完全接收到Client傳送的所有訊息, 且最終能依序讀取並顯示收到的封包。

      • 基於HW5的經驗，藉由設計不同delay, loss參數的Proxy，產生不同的網路端對端傳出效能提供給同學觀察。
      • 請同學設計新的Client傳輸封包給Server的方法，改善原本HW5傳輸方式所得到的效能。

實驗拓樸

(a) Socket 架構

  • 為提供方法設計與開發彈性， Proxy 提供兩個 port 類比網路中兩條不同的路徑支援Client與Server間傳輸。
  • 實作時可根據需求選擇其中一條或兩條路徑進行傳輸。
  • Client必須透過UDP proxy將封包傳給Server。

(b) 硬體拓樸

  • 連線設定與HW5的p.8相同，唯一差異僅在PC1的UDP proxy具有兩個port，而HW5只有一個proxy提供給Client進行傳輸。

目標:

1. 透過課程提供三個版本的Proxy (如下述)，以及同學們改良HW5的Client & Server，將拓樸的架構實作出來

2. 觀察Client傳送與Server接收封包的效果，並著手思考並實作可能的改良傳輸效能的方法。

3. 輸出結果要求

  • 封包按照順序在Server端顯示

  • 不可重複, 亂序, 缺少任一封包

  • 計算Client完整傳輸所有封包給Server所需的時間

(35%) Proxy版本一:

  1. Proxy 的行為

    • 有 5% 機率發生 packet loss

  2. Proxy 的設置:

    • IP address: 192.168.2.100

    • Input port: 5405 或 10001 (可同時使用)

    • Terminal 的結果: 僅會顯示兩個 port 是否已經開啟

      備註: 由於 Proxy 是透過 threading 設計的, 因此無法透過 ctrl + c 中斷, 只能主動將 terminal 關閉以結束 Proxy.

      • Wireshark 或 Terminal 的結果皆可

      • 請務必提供使用的資料來源
    2. 清楚地描述設計方法.

改良參考範例 (此場景僅供概念參考):

  • 機制:

    • 採用概念: 冗餘鏈路

    • 如果至少有一條路徑有成功送達 Client, 則保持輸出結果

      • 兩條鏈路都失敗的機率很小: 0.1∗0.1=0.001=0.1%
      • 因此使用雙連線成功傳輸的機率會較高

    • Server 透過綠色路徑傳遞當前接收狀態給 client.

    • 如果 Loss 和 Delay 同時發生, 則通知 Server 重新傳送該封包.

      • 此處設計的重新傳送如果沒有遇到 Loss / Delay, 將於 0.1 秒後完成, 會比等待該封包自行回復更有效率.
